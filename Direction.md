# SysML v2 標準用語集（Glossary）策定ガイドライン (draft)

## 1. プロジェクトの背景と現状
SysML v2 仕様書（約1,500ページ）から、LLM（Gemini）を用いて専門用語の抽出・一次翻訳を行いました。
現在、**正味 1,061 件**（延べ 1,303 件）のドラフトデータが揃っています。このデータの精度は学術的に中立で高く、約 60〜80% はそのまま実戦投入可能な品質です。

本プロジェクトのゴールは、この 1,061 件を **「チーム標準」として 700〜800 件程度に厳選・固定化** することです。

---

## 2. 精査の基本戦略： 80対20の法則
全件を議論の対象にすると、不必要に多大な時間を消費してしまいます。
以下の3ステップで、リソースを「真に議論が必要な箇所」に集中させます。

### STEP 1：ノイズの「一括選別」（マイナス作業）
* **内容**: 1,061件から不要な項目を削ります。
* **削除基準**:
    * 一般的なIT用語（String, Boolean, Real等）
    * 単なる動詞（calculate, identify等）
    * 章の導入部のみに現れる比喩的表現
* **目標**: 全体を **約700件** までスリム化します。
* **方針**: 削除作業は、手分けして行い、削除対象の見積もり300件程度は全員で共有してレビューします

### STEP 2：AIドラフトの「サイレント承認」（80%の処理）
* **内容**: 「Geminiの訳で問題ない」ものを議論なしでパスさせます。
* **方式**: ネガティブ・チェック（違和感があるものだけを「要議論」にマーク）。
* **目標**: 議論対象を **300件以下** に絞り込みます。
* **方針**: 複数人でネガティブチェックを行い、一人でもチェックが付いたものはSTEP3の対象とします

### STEP 3：3人1組の「トリオ・レビュー」（20%の濃縮議論）
* **内容**: 残った「要確認」の項目を、3チーム（3〜4名構成）で並行レビューします。
* **審拠**: 主観ではなく、データに含まれる `Definition`（定義）と `Context/Example`（文脈例）を唯一の判断材料とします。

## 4. 審議における「黄金のルール」
議論を迷走させないため、以下の優先順位で訳語を確定させます。

1.  **公的規格優先**: JIS/ISO等で既定されている単位系や数学用語は無条件採用。
2.  **文脈優先**: 仕様書の `Definition` 文中で、他の用語とどう関連しているかを重視。
3.  **エンジニアの洞察**: 実際にモデリングを行う際に、意味が直観的にわかる言葉を選ぶ。
4.  **日本語としての妥当性**: 後世に残る日本語の用語としてふさわしい言葉を選ぶ。

## 5. 使用データ（CSV）の構成
マスターデータには以下の列が含まれています。

* **Term (EN)**: 英単語（一意キー）
* **Equivalent (JA)**: Geminiによる一次訳（精査対象）
* **Definition (EN)**: 仕様書上の定義（審議の根拠）
* **Context/Example (EN)**: 出現箇所のコード例や補足
* **Section**: 仕様書の章番号

これを下に以下の列を加えて、作業用ファイルを構成します。
編集はExcelなど好きなCSVエディタで可能なように配慮されています。

* **Status**: [OK / 削除 / 要議論]（各自で更新）

各々の編集後、都度、CSVの正規化をPython CSVにて行うことで、適切に差分が確認できるようにします。
